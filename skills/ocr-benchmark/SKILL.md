# OCR Benchmark

把 OCR 评测作为可复现的数据流程执行。保留原始证据，隔离不同参数组，只发布核验后的结果。

## 首次使用先确认裁判配置

首次在当前评测任务中使用本 skill 时，先询问用户：

1. 裁判模型的 `model id` 是什么？
2. 裁判 API 的 `URL` 是什么？

即使仓库内存在默认值，也不要擅自采用。若用户已在当前会话明确给出两项配置，复述确认即可，不重复提问。API key 只通过当前进程环境变量接收，不写入 skill、脚本、日志或文档。

## 建立本次评测配置

读取仓库 `AGENTS.md`、评测规则和现有脚本，然后确认：

- 测试数据入口；
- 可选 manifest、预期样本数和数据集版本；
- 参赛模型 ID、API URL、来源标签和 thinking 模式；
- 提示词、采样参数和并发数；
- 原始运行输出目录；
- 报告输出目录；
- 排行输出目录；
- 裁判 model id、API URL、提示词版本和 reasoning 模式。

测试集不得写死。用户可以传入 JSONL、JSON、目录或仓库支持的其他入口；若未指定，检查仓库后列出候选入口并请用户确认。报告目录和排行目录必须作为两个独立配置处理，也允许只生成其中一种产物。

需要命令模板和配置字段时，读取 [references/workflow.md](references/workflow.md)。

需要新建、迁移或整理测试集时，读取 [references/testset-template.md](references/testset-template.md)。该模板提供最小结构和带分类、难度、标签、冻结审计的扩展结构；目录名和字段均可按项目规则替换。

## 默认提示词

提示词优先级固定为：用户明确指定 > 当前仓库 benchmark 规则 > skill 默认提示词。只有前两项都不存在时才使用：

- [OCR 默认提示词](assets/default-prompts/ocr.txt)
- [LLM 裁判默认提示词](assets/default-prompts/judge.txt)

使用默认提示词时，在运行配置和报告中明确标记。一次正式运行开始后不得静默改词；修改提示词必须创建新的提示词组。

## 内置脚本

- [scripts/init_testset.py](scripts/init_testset.py)：按最小或扩展模板规划、创建测试集骨架；默认 dry-run，`--apply` 后才写入。
- [scripts/audit_testset.py](scripts/audit_testset.py)：检查 manifest、文件配对、空标注、重复 ID/图片、声明哈希和可选训练集重叠。
- [scripts/validate_run.py](scripts/validate_run.py)：比较测试入口、预测和裁判 JSONL 的唯一记录、缺失、额外记录与空输出。

先运行脚本的 `--help`。仓库已有等价脚本时优先使用仓库版本；内置脚本作为新项目起步和跨仓库兜底。

## 运行模型推理

1. 优先复用仓库评测脚本，不重复实现客户端。
2. 先用 1 个代表样本验证视觉输入、提示词位置、参数兼容性、返回解析和 usage 字段。
3. 使用稳定文件名保存 JSONL，按已完成图片断点续跑；统计唯一图片数，不把错误尝试次数当作缺失样本数。
4. 从保守并发开始。出现成组 502/503/504 后降低并发；只有连续成功时才提高并发。
5. 不擅自增加提示词、后处理、缩图或模型专用参数。
6. 用户要求改变参数补跑时，隔离失败样本并记录逐样本覆盖值；不得把混合参数结果描述成统一参数结果。

持续失败的请求保留为缺失记录。不得伪造预测，也不得静默换用其他模型、提供方、参数组或测试集结果。

## 统一执行裁判

1. 裁判前核验预测数、唯一图片数、空输出、模型 ID 和实际请求提示词。
2. 使用用户确认的裁判 model id 和 API URL，以及当前 benchmark 定义的裁判提示词和响应 schema。
3. 正式裁判保持 model id、reasoning 模式和提示词版本一致；诊断裁判与争议复核单独保存。
4. 服务稳定时使用 2–5 并发。遇到 `model_not_found`、额度耗尽或系统性格式错误时立即停止，不继续制造无效请求。
5. 短暂错误续跑同一裁判 JSONL。除非用户明确创建新评分组，否则不得在一个正式分数中混合裁判模型。

## 计分与核验

使用仓库提供的计分脚本，并以本次选择的完整测试入口或 manifest 为基准。至少核验：

- 预期样本数；
- 预测和裁判图片唯一；
- 意外空输出为 0；
- 缺失记录显式计入；
- 正式裁判 model id 和提示词版本唯一；
- 分类、难度、severe、完成率、安全分、NED 和可靠性指标齐全；
- 输出和文档不含 API key。

不得用局部均值推测最终分；缺失项和风险扣分必须由正式计分流程处理。

## 生成报告

把报告写入用户指定的报告目录。先说结论，再写真实优势和失败方式，把分类、难度、标签和 badcase 明细放到底部。避免模型性格、自我意识、空泛总结和无证据因果。

报告必须注明模型名、thinking、API 来源、测试集、提示词组、关键参数、记录数、缺失原因、裁判配置和最终指标。若未要求报告，不要自动生成。

仓库没有既定格式时，复制并填写 [报告模板](assets/templates/report.md)。不要在模板原文件上直接写评测结果。

## 更新排行

只在用户要求时更新排行，并写入单独指定的排行目录。新增模型后重新计算排序以及分类、难度最佳项；不要只插入一行而保留过期结论。验证报告链接存在，再比较源文件与目标文件哈希。不得覆盖无关排行或报告。

仓库没有既定格式时，复制并填写 [榜单模板](assets/templates/ranking.md)。不同测试集、提示词组或不兼容的裁判口径分表展示，不强行混排。

## 交付

汇报最终结果、缺失记录、补跑覆盖参数、关键文件路径，以及报告和排行各自是否生成。把服务端原因写成基于证据的推断，不写成已证实事实。结束前确认没有意外遗留的评测进程。
