# 开发场景 OCR Benchmark v4

更新时间：2026-06-13

传统 OCR benchmark 往往只看字符级相似度，但开发场景 OCR 的实际风险并不等同于字符编辑距离。代码、配置、终端、错误日志和 IDE 截图更关心符号、缩进、阅读顺序、结构格式、噪声控制，以及开发者能否直接复制、搜索和理解输出文本。

Benchmark v4 在 v3 的 LLM 六维裁判经验上扩展数据集并重做最终积分公式：测试集从 50 题扩展到 100 题，并按活动规则覆盖 8 类开发场景。v4 结果不能与 v3 50 题结果直接混排。

## Benchmark v4 结果

按 `benchmark_guide_v4.md` 规则从 LLM 六维评分结果重算。最终排序使用分类加权后的 `final_score_v4`。平均 LLM 分和 NED 仅展示，不直接参与最终加权。为避免提示词策略混排，短提示和长/严格提示分开排名。

诊断指标说明：

- 严格可用率：LLM 总分 `>75` 且噪声与可用性维度 `>6/10` 的样本比例，近似表示“基本可直接复制、搜索或参考”的比例。
- 完成率：噪声与可用性维度 `>1/10` 的样本比例，近似表示模型至少完成了有效 OCR 输出，而不是空输出、拒答、严重跑题或完全不可用。
- 安全分：`1 - severe badcase rate`，severe 包括严重漏识别、结构崩坏、幻觉/重复、可执行代码被增删改写、关键维度极低等高风险样本；越高表示灾难性错误越少。
- 类别均衡分：基于 8 类 `category_score` 的诊断分，公式为 `clamp(0.4 * mean + 0.6 * median - 0.5 * range - 0.25 * abs(mean - median), 0, 100)`；其中 `range` 是最高类别分与最低类别分分差。该项不参与最终积分。
- 思考模式：仅标注模型调用形态，不参与评分；Qwen 百炼按 `thinking` 记录，GPT-5.5 记录实际 `reasoning_effort`，PaddleOCR-VL 本地/API 为非思考 OCR 解码，Seed2 Lite 按 `medium` 调用记录。

## LLM 六维评分

裁判版本为 `dev_ocr_judge_v4`。裁判只比较人工真值 `label` 与模型输出 `prediction`，不把 NED 传入 prompt。

| 维度 | 满分 | 含义 |
| --- | ---: | --- |
| `content_coverage_0_10` | 10 | 主要内容覆盖 |
| `symbol_accuracy_0_10` | 10 | 字符、代码符号、大小写、数字、路径、标点准确性 |
| `indentation_alignment_0_10` | 10 | 缩进、嵌套层级、对齐 |
| `structure_format_0_10` | 10 | 换行、代码块、列表、表格、区域结构 |
| `reading_region_order_0_10` | 10 | 阅读顺序、区域顺序、多文本块顺序 |
| `noise_and_usability_0_10` | 10 | 重复、幻觉、无关文本、解释性文字控制和开发者实际可用性 |

## 积分规则

| 项 | 权重 | 计算方式 |
| --- | ---: | --- |
| 内容覆盖 | 20% | 类别内 `content_coverage_0_10 / 10` 平均值的 `1.4` 次方。 |
| 符号准确性 | 24% | 类别内 `symbol_accuracy_0_10 / 10` 平均值的 `1.4` 次方。 |
| 缩进与对齐 | 16% | 类别内 `indentation_alignment_0_10 / 10` 平均值的 `1.4` 次方。 |
| 结构格式 | 14% | 类别内 `structure_format_0_10 / 10` 平均值的 `1.4` 次方。 |
| 阅读顺序 | 10% | 类别内 `reading_region_order_0_10 / 10` 平均值的 `1.4` 次方。 |
| 噪声与可用性 | 16% | 类别内 `noise_and_usability_0_10 / 10` 平均值的 `1.4` 次方。 |

最终公式：

```text
category_score =
100 * (
  0.20 * content_coverage^1.4
+ 0.24 * symbol_accuracy^1.4
+ 0.16 * indentation_alignment^1.4
+ 0.14 * structure_format^1.4
+ 0.10 * reading_region_order^1.4
+ 0.16 * noise_and_usability^1.4
)

final_score_v4 = sum(category_weight * category_score)
```

分类权重：

| 类别 | 权重 |
| --- | ---: |
| IDE 代码编辑器截图 | 15% |
| 终端 / Shell / PowerShell | 7% |
| 报错日志 / Traceback | 17% |
| 配置文件 / YAML / JSON / TOML / INI | 14% |
| Git diff / patch / PR 页面 | 7% |
| 网页代码块 / 文档代码块 | 10% |
| 表格化代码信息 / 参数表 / API 文档 | 12% |
| 模糊、压缩、拍屏、暗色主题、小字号等困难样本 | 18% |

严格可用率、完成率、安全分、severe、错误标签和类别均衡分只作为诊断指标，不参与最终分。

## Severe Badcase

以下情况会被视为 severe badcase：

```text
missing_record
or sample_fidelity < 0.35
or noise_and_usability_0_10 <= 1
or score_0_100 < 40
or any(front_five_dimension_ratio < 0.2)
or error_tags contains code_added_removed_or_rewritten
```

如果某个模型在任一类别中 severe 比例超过 40%，即使整体分不低，也应在报告中标注为该类别不稳定。
