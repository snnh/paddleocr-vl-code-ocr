# PaddleOCR-VL 开发场景代码 OCR 训练数据构建报告

整理日期：2026-06-19  
提交阶段：初赛提交版  
任务方向：代码文字识别 / 开发场景 OCR

## 1. 任务定义

本项目基于 PaddleOCR-VL-1.6 微调开发场景代码 OCR 模型。输入为 IDE、终端、Traceback、配置文件、Git diff、文档代码块、API 表格、小字号和拍屏等开发相关图片；输出为图片中可见文字。

统一训练与推理提示词：

```text
<image>OCR:
```

输出要求是只转写可见文本，不解释、不总结、不补全不可见内容，并尽量保留代码符号、大小写、缩进、换行、路径、文件名和阅读顺序。

## 2. 场景价值

普通 OCR 在开发场景中容易出现符号识别错误、缩进丢失、结构错乱、解释性包装、不可见内容补全和代码改写。本项目聚焦以下长尾能力：

- 代码符号保真：括号、引号、冒号、逗号、等号、箭头、路径分隔符等。
- 结构保真：Python/YAML/Markdown/树状结构中的缩进、层级和换行。
- 多区域顺序：IDE 标签页、正文、终端、错误面板、Diff 和 API 表格的阅读顺序。
- 抗噪声：小字号、暗色主题、压缩、拍屏、局部模糊和多区域混排。
- 可用性：减少幻觉、解释性文字、Markdown 包装和擅自补全。

- 训练集、过程验证集、旧测试集和 benchmark v4 测试集按 stem 与图片哈希去重。

## 4. 数据采集方法

数据来源包括：

- IDE、编辑器、终端、Notebook、调试器、SSH 工具和代码工具截图。
- 公开代码、配置、README、API 文档、错误日志和 Git diff 的截图或网页元素截取。
- CodeSCAN 等公开代码截图数据。
- 针对 benchmark badcase 的定向候选池，包括 YAML/JSON/TOML/INI、API 表格、Markdown 表格、ASCII 诊断、短输出陷阱、多区域 IDE、长 Traceback 和符号密集代码。
- 小字号、暗色主题、压缩、拍屏、局部模糊等困难样本增强。

关键脚本和工具：

| 脚本/工具 | 用途 |
| --- | --- |
| `scripts/collect_real_code_ocr_samples.py` | 采集真实或公开开发场景代码 OCR 候选样本 |
| `scripts/collect_diverse_ide_screenshots.py` | 构建多主题、多 IDE 风格候选截图 |
| `scripts/build_v5_badcase_candidates.py` | 根据阶段性 badcase 定向生成补充候选 |
| `scripts/audit_dataset_quality.py` | 检查缺失文件、空文本、重复和明显格式问题 |
| 人工审核流程 | 对 OCR 初稿和候选样本进行人工转写、剔除和复核 |

## 5. 标注规范

- OCR 真值以人工 `.txt` 或 `train.json` 中 assistant 内容为准。
- 保留原始换行、缩进、大小写、标点、括号、引号、运算符、路径和数字。
- 文件名、标签页、路径与正文之间可用单独一行 `---` 分隔。
- Markdown 样本保留标题、列表、代码围栏和表格结构。
- 编辑器左侧纯行号通常不作为真值内容，除非行号是截图主体语义的一部分。
- 不补全图片外不可见内容，不把代码改写成等价代码。
- 含 `[待人工转写]`、空标注、明显幻觉或无法辨认主体文本的样本不得合入训练。

## 6. 标注工具与流程

```text
候选图片
  |
规则初筛与去重
  |
OCR/VLM 生成初稿或人工直接转写
  |
人工复核正常可见文本
  |
剔除低质、无关、重复和占位样本
  |
导出 train.json 与 benchmark/test 索引
  |
训练回归与 badcase 分析
```

人工复核重点：

- 主体文本是否完整。
- 符号、缩进、换行、路径和大小写是否正确。
- 是否混入解释性文字、不可见补全或模型幻觉。
- 图片与标注是否一一对应。
- 是否与 benchmark v4 发生重复。

## 7. 质量控制

已采用的质量控制措施：

- 图片与文本配对检查。
- 空标注、占位标注、emoji 残留和明显异常输出检查。
- stem 与图片哈希去重。
- benchmark v4 固定后不根据测试结果修改真值。
- 评估集新增样本保留来源审计、图片哈希和标注哈希。
- badcase 单独统计，包括严重漏识别、结构崩坏、重复、幻觉和代码改写。

评估集真实性控制：

- benchmark v4 为 100 题冻结测试集，包含 8 类开发场景。
- 新增 50 张评估样本中 `real_web_element` 32 张、`github_rendered_code_ui` 8 张、`rendered_diff_or_patch_code_block` 10 张。
- 所有评估样本均以人工真值为准，相关记录见 `评估集正式提交/code_ocr_eval_dataset_v1/audit/`。

## 8. 微调策略

当前提交候选为 PaddleOCR-VL-1.6 微调 v6。

训练策略：

- 基础模型：PaddleOCR-VL-1.6。
- 微调方式：LoRA。
- prompt：`<image>OCR:`。
- 训练目标：只输出图片中可见文字。
- 数据策略：在真实截图和公开样本基础上，加入针对符号、缩进、配置、Traceback、Diff、API 表格和困难图像的 badcase 定向补充。
- 回归策略：固定 benchmark v4 100 题，比较 NED、LLM 六维分、严重 badcase 和可用率。

推荐推理参数：

```text
max_tokens=4096
repetition_penalty=1.08
temperature=0
```

benchmark v4 使用 100 题冻结评估集和 `<image>OCR:` 短提示。v6 当前结果：

| --- | ---: |
| final_score_v4 | 61.08 |
| 全局六维分 | 64.06 |
| 类别宏平均 | 63.86 |
| 最弱类别 | 44.54 |

v6 是当前本地最高结果，超过 v5-v2params 的 `57.85` 和 v2 的 `55.55`。后续回归以 v6、`max_tokens=4096`、`repetition_penalty=1.08`、`temperature=0` 作为主基线。

## 10. 已知限制

当前模型仍可能在以下场景出错：

- 极小字号、模糊或压缩严重图片。
- 复杂 API 表格和多列表格。
- 深层嵌套配置文件。
- 长 Traceback 和多区域错误日志。
- 罕见符号、路径、转义字符和缩进敏感代码。
- 真实终端或 IDE 截图中混入大量 UI 小字时，可能漏转写或顺序混乱。

下一阶段将继续补充高质量真实截图和定向 badcase，优先提升困难样本、配置文件、Traceback 和 API 表格类别，而不是单纯增加训练 epoch。
