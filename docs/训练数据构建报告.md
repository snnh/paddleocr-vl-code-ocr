# PaddleOCR-VL 开发场景代码 OCR 训练数据构建报告

整理日期：2026-06-13  
提交阶段：初赛公开版

## 1. 任务定义

本项目面向开发场景 OCR，目标是基于 PaddleOCR-VL-1.6 微调一个更适合识别代码和开发工具文本的模型。输入为 IDE、终端、Notebook、调试器、错误列表、补全框、文件树、配置文件、Git diff、文档代码块等截图，输出为图片中可见文字。

统一训练与推理提示词：

```text
<image>OCR:
```

## 2. 场景价值

普通 OCR 在代码场景中容易出现符号错误、缩进丢失、结构错乱、解释性输出、不可见内容补全和代码改写。本项目重点提升以下能力：

- 括号、引号、冒号、逗号、等号、箭头、路径分隔符等代码符号准确性。
- Python/YAML/Markdown/树状结构中的缩进与层级保真。
- 终端、多区域 IDE、Traceback、Diff 和 API 表格中的阅读顺序。
- 小字号、暗色主题、压缩和拍屏样本中的抗噪声能力。
- 严格只输出图片中可见文本，减少幻觉和解释性包装。

## 3. 数据集状态

训练数据仍在清洗、补充和冻结中，本公开仓库仅说明数据构建原则。数据切分原则如下：

| 切分 | 用途 |
| --- | --- |
| 训练集 | 用于 PaddleOCR-VL-1.6 LoRA 微调，持续迭代中。 |
| 过程验证集 | 用于训练过程观察，不作为最终 benchmark。 |
| benchmark v4 测试集 | 冻结测试集，不参与训练和训练期调参。 |

约束：

- benchmark v4 测试集冻结，不参与训练和训练期调参。
- 训练索引与测试索引保持无重叠。
- 旧 50 题测试集已归档，不与当前 100 题 benchmark 混用。

## 4. 数据采集方法

数据来源包括：

- IDE、编辑器、终端、Notebook、调试器和代码工具截图。
- 开源代码、配置文件、Markdown 文档、API 文档和错误日志渲染截图。
- 面向 badcase 的定向候选池，包括 YAML/JSON/TOML/INI、API/Markdown 表格、ASCII 诊断、短输出陷阱、多区域 IDE 和符号密集代码。
- 拍屏、压缩、小字号和暗色主题等困难样本增强。

关键代码：

| 脚本 | 用途 |
| --- | --- |
| 采集脚本 | 从真实开源仓库、文档和开发工具页面采集或渲染候选截图。 |
| 初筛脚本 | 使用规则和 VLM 辅助判断图片是否适合开发场景 OCR。 |
| badcase 候选生成脚本 | 根据阶段性 badcase 画像生成定向补充候选池。 |
| `dataset/audit_dataset_quality.py` | 检查缺失文件、空文本、重复样本和明显格式问题。 |

## 5. 标注规范

- OCR 真值以人工 `.txt` 为准。
- 尽量保留原始换行、缩进、大小写、标点、括号、引号、运算符和路径。
- 文件名、标签页、面包屑路径与正文之间用单独一行 `---` 分隔。
- Markdown 样本保留标题、列表和代码围栏。
- 编辑器左侧纯行号通常不作为真值内容。
- 禁止补全图片中不可见内容。
- 含 `[待人工转写]`、空标注、emoji 或明显模型幻觉的样本不得合入训练。

## 6. 质量控制流程

```text
候选截图
  |
去重、尺寸/格式检查、开发场景过滤
  |
模型生成 OCR 初稿
  |
人工复核与质量审计
  |
导出 train/test 索引
  |
本地 benchmark 与 badcase 分析
```

质量控制措施：

- 图片与 `.txt` 一一配对。
- 主集之间进行同名 stem 和图片哈希去重。
- 测试集固定后不根据测试结果修改真值。
- 弱标注不直接作为最终真值，进入训练前需人工快审。
- 长输出发散、重复、幻觉、空输出和高 NED 样本单独整理为 badcase。

## 7. 模型微调策略

当前训练仍在迭代中。公开仓库只保留策略摘要，不放完整训练 notebook。

- 基础模型：PaddleOCR-VL-1.6。
- 微调方式：LoRA。
- 训练提示词：`<image>OCR:`。
- 输出目标：只输出图片中可见文字。
- 参数：保留阶段性参数摘要，最终参数以模型卡发布版本为准。
- 推理推荐：`max_tokens=4096, repetition_penalty=1.10, temperature=0`。

阶段性 benchmark 仅用于内部回归，不作为最终模型性能声明。

## 8. Benchmark 与反馈闭环

本项目使用 benchmark v4 作为当前主要回归口径。评估集包含 100 题，按 8 类活动权重汇总。LLM 裁判从六个维度评分：

- 主要内容覆盖。
- 字符与符号准确性。
- 缩进与对齐。
- 结构格式。
- 阅读/区域顺序。
- 噪声与可用性。

最终目标不是单纯降低 NED，而是提高开发者可用性，减少严重 badcase、符号错误、结构破坏和擅自补全。

## 9. 已知限制与下一步

当前模型仍弱于强外部 VLM，主要短板包括：

- 困难样本。
- 表格化 API 文档。
- 配置文件复杂层级。
- 长 Traceback 和多区域结构。
- 错符号、漏主体文本、缩进破坏、幻觉和代码改写。

下一步计划优先补充高质量 badcase 定向数据，不直接盲目增加 epoch。
