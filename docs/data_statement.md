# 数据集说明

## 数据来源

本评估集围绕真实开发工作流构建，覆盖 IDE、终端、配置、Traceback、Git diff、文档代码块、API 表格和困难样本。数据来源包括：

- 真实或半真实开发场景截图。
- 开源代码、配置、文档和错误信息片段的渲染截图。
- 为覆盖特定弱项而构造的 UI/代码渲染样本。
- 暗色主题、小字号、压缩、拍屏和多区域混排等增强样本。

评估集中的目标不是普通文档 OCR，而是代码和开发工具文本 OCR。样本选择强调符号密度、缩进层级、结构复杂度和实际可用性。

## 标注规范

- 每张图片对应同名 `.txt` 人工真值。
- 尽量保留原始换行、缩进、大小写、标点、括号、引号、运算符和路径。
- 文件名、标签页、面包屑路径与正文之间可使用单独一行 `---` 分隔。
- Markdown 样本保留标题、列表和代码围栏。
- 编辑器左侧纯行号通常不作为真值内容。
- 禁止补全图片中不可见的内容。

## 切分与泄漏控制

- 当前评估集为 100 题冻结测试集。
- 评估集不参与训练和训练期调参。
- 新增样本与 `train/`、`评估集_20260601/` 和旧测试集做了 stem 与图片哈希去重。
- 相关质检记录存放在 `audit/`。

## 类别与难度

评估集按比赛任务价值和开发 OCR 难点拆成 8 类。高权重难点包括：

- Traceback 和多行错误日志中的阅读顺序与结构保持。
- YAML/JSON/TOML/INI 等配置文件中的层级和符号准确性。
- API 表格和参数表中的字段、列、代码片段和说明文字对齐。
- 小字号、暗色主题、压缩、拍屏和多区域混排样本中的抗噪声能力。

## 合成与渲染说明

赛事规则特别强调评估集真实性，若合成数据占比过高会影响评估集质量甚至触发一票否决。本评估集包含一部分为覆盖长尾弱项而生成或渲染的 UI/代码截图，但正式提交版保留逐样本来源审计，并且所有提交样本均以人工真值为准，经过人工核对，不使用 OCR/VLM 自动输出直接作为最终标注。

当前 100 张冻结评估集由两部分组成：

- 早期冻结测试池 50 张：来自真实或半真实开发场景截图、开发工具界面截图和困难样本整理，已完成图片与标注配对、人工复核和泄漏检查。
- 新增补充样本 50 张：带有来源审计记录，其中 `real_web_element` 32 张、`github_rendered_code_ui` 8 张、`rendered_diff_or_patch_code_block` 10 张。

新增 50 张中，网页真实元素占 64%，GitHub/代码 UI 渲染占 16%，diff/patch 代码块渲染占 20%。这些渲染样本主要用于补齐 Git diff、配置文件、文档代码块、API 表格和困难样本等长尾场景，不作为训练集使用，也不参与训练期调参。

来源审计文件为 `audit/sources.jsonl`，其中记录文件名、类别、来源 URL、采集类型、图片哈希和标注哈希。
